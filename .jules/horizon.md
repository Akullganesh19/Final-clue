## 2026-07-01 — Cryptographic Hashing and UUID Generation
**Risk identified:** The legacy audit logic was generating identifiers using `Date.now()` and `Math.random()`, which can lead to collisions and are not cryptographically secure. The hashing algorithm was a custom bitwise operation, which is vulnerable and no longer aligned with modern web standards. If left unchecked, this could lead to integrity compromises in the audit trail, costing much more to remediate later when the data volume scales up.
**Migration target:** The modern Web Crypto API (`crypto.randomUUID()` for unique IDs and `crypto.subtle.digest('SHA-256')` for robust hashing) which provides native, secure cryptographic operations that conform to ecosystem standards.
**Migrated this session:** `generateAuditHash` and `createAuditLog` in `src/utils/audit.ts` were updated to use `crypto.randomUUID()` for robust identifier generation. An additive migration path (`generateAuditHashAsync` and `createAuditLogAsync`) was introduced to implement asynchronous cryptographic hashing using `crypto.subtle.digest('SHA-256')` without breaking existing synchronous consumers.
**Remaining:** Migrate existing synchronous call sites of `createAuditLog` to use the new `createAuditLogAsync` and correctly handle promises, ensuring UI gracefully manages asynchronous state updates for logs.
**Next session:** Investigate components rendering or producing audit trails (e.g. forms or event emitters calling `createAuditLog`) and incrementally update them to use `createAuditLogAsync` along with any necessary Optimistic Concurrency Control (OCC) parameters.
