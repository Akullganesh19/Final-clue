## 2024-05-15 — [PII Leakage in Audit Logs]
**Data traced:** PII data (Emails, SSNs, phone numbers, and credit cards).
**Exposure found:** Plaintext PII strings were actively being collected via parameters `action`, `details` and `author` inside `createAuditLog` and subsequently saved unmasked to the blockchain-style `AuditTrail` store that persists the app's overall actions and state. The parameters used an unsafe delimiter format `|` which could lead to delimiter injection vulnerabilities.
**Fix:** Introduced an irreversible regex-based redaction utility `redactPII` to mask sensitive data while preserving debug context (e.g., `j***@gmail.com`). This utility was integrated structurally directly into `createAuditLog` so all incoming audit details are forcefully redacted before generating the CHK hash or appending logs to memory. The parameter delimiter logic in `generateAuditHash` was converted to `JSON.stringify` serialization.
**Coverage confirmed:** Tested using native Node.js testing module via `npm test`. Wrote tests simulating active exposure points verifying phone numbers, SSNs, cc numbers, emails mask perfectly into the `createAuditLog` outputs.
**Still exposed elsewhere:** The current redaction utility runs globally over the action, details, and author fields using heuristics (regex); there could still be edge-case formats of un-structured text where a highly obfuscated PII leaks. There might also be other un-checked log-sinks, analytics, or caching layers where raw un-redacted payload parameters are logged manually before they reach `createAuditLog`.
